## Notification Management Overview

Notification Management in the TEO KICKS API system is responsible for creating and delivering various types of alerts and updates to users. Notifications are typically triggered by system events (e.g., order status changes, payment success) and can be delivered in-app via WebSockets. Users can manage their preferences for different notification channels (email, SMS, in-app) through user profile settings.

---

## 👤 Notification Model

### Schema Definition
```typescript
interface INotification {
  _id: string;
  userId: string; // User ObjectId
  type: "order_created" | "payment_success" | "order_status_changed" | "invoice_generated" | "receipt_issued";
  payload?: any;
  read: boolean;
  createdAt: Date;
  updatedAt: Date;
}
```

### Model Implementation

**File: `../models/notificationModel.js`**

```javascript
import mongoose from "mongoose"


const notificationSchema = new mongoose.Schema({

    userId: { type: mongoose.Schema.Types.ObjectId, ref: "User", required: true },
    type: {
        type: String,
        enum: [
            "order_created", "payment_success", "order_status_changed", "invoice_generated", "receipt_issued"
        ],
        required: true
    },
    payload: { type: mongoose.Schema.Types.Mixed, default: {} },
    read: { type: Boolean, default: false }

}, { timestamps: true })


notificationSchema.index({ userId: 1, createdAt: -1 })
notificationSchema.index({ type: 1, createdAt: -1 })


const Notification = mongoose.model("Notification", notificationSchema)


export default Notification
```

### Validation Rules
```javascript
userId:    { required: true, type: ObjectId, ref: 'User' }
type:      { required: true, type: String, enum: ['order_created', 'payment_success', 'order_status_changed', 'invoice_generated', 'receipt_issued'] }
payload:   { type: Mixed, default: {} }
read:      { type: Boolean, default: false }
```

## 🎮 Notification Controller

**Not Applicable:** There is no dedicated `notificationController.js`. Notifications are typically created by internal services or other controllers (e.g., `orderController`, `paymentController`, `authController`) and delivered via the internal `notificationService.js` (located at `server/services/internal/notificationService.js`) and Socket.io.

**Notification Creation Flow:**
- **Authentication Events:** `authController.js` uses `notificationService.js` to send OTP, password reset, and welcome notifications via email and SMS.
- **Order Events:** `orderController.js` may trigger order status notifications through the notification service.
- **Payment Events:** `paymentController.js` may trigger payment success/failure notifications.
- **Delivery:** Notifications are delivered via WebSockets (Socket.io) for real-time in-app notifications, and via email/SMS through external services (`emailService.js`, `smsService.js`).

User preferences for notification channels are managed through `userController.js`.

## 🛡️ Security Features

-   **Implicit Access Control:** Notifications are linked to specific `userId`s, ensuring that a user only receives and can access their relevant notifications.
-   **System-Generated:** Notifications are primarily generated by trusted backend services in response to business logic, reducing the risk of malicious notification creation.
-   **User Preferences:** Users have control over which types of notifications they receive via different channels (email, SMS, in-app), managed through secure user profile endpoints.

---

## 🚨 Error Handling

Errors related to notifications would typically occur during their internal generation or delivery process, managed by the relevant services (e.g., `notificationService.js`). Unhandled errors in this domain would likely manifest as `500 Internal Server Error` if a system event fails to trigger a notification.

## 📊 Database Indexes

-   `userId: 1, createdAt: -1`: For efficient retrieval of a user's notifications, ordered by the most recent first.
-   `type: 1, createdAt: -1`: For efficient querying of notifications by their type (e.g., all `order_status_changed` notifications), sorted by creation date.
